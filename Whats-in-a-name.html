Why are labels important?

Before we talk about how to make labels accessible, let's talk about why labels are important.

I've included an extreme example here of five input fields with no labels whatsoever.

It's possible that based on the context of this form, I could find a way to fill in all this information correctly. But it's more likely that I'll be confused, not know what I should enter, and make mistakes.

I might abandon this form altogether and decide it's not worth the trouble.

Now this is an intentionally extreme example, and it's not something I've ever seen on the web. But the examples I'll show you next are all things I've seen some variation of before, and they all make filling out forms difficult, unusable, and inaccessible form some users.

Labels must be visible

Date

Our first rule of labelling is that labels must be visible.

It might seem obvious that all form fields must have visible labels. As I said, I almost never see form fields without labels in isolation. What I do see is ambiguous fields.

In this example, I have three input fields under one date label. You might assume that the three fields are for day, month, and year. But depending on where you are in the world, day and month might be the first and second field respectively or the other way around.

This form isn't clear to anyone, and it will be very easy to input the incorrect date.

Labels must be clear

The next rule is that labels must be clear.

In this example, there are two input fields, both with the label name.

My assumption would be that one of these is given name and the other is family name, but I can't be certain which is which. This also depends on where in the world this form is coming from.

And it might be something else entirely. It might be legal name and maiden name or nickname. The labels aren't clear enough to be sure.

So when you're creating labels, always make sure that they clearly identify what information is being requested.

Labels must persist

Placeholders won't cut it!

Contact Information

Next we have labels must persist and placeholders won't cut it!

Here we have a contact information form with two fields and a submit button. Neither field has a label. The second field says Email in placeholder text. The first field just has the letter "A."

This is because I've already started filling out this field. But once I fill in information, the placeholder text disappears.

If I had short-term memory loss, or I was distracted by my two year old, or had a big headache today, I might not remember what the placeholder text said.

I would have to delete the information to see the placeholder again. And if I also had pain in my hands or limited mobility in my hands, that might be painful or more difficult.

There are other issues with placeholder text as well:

  • Placeholder text tends to be lighter than surrounding text. This can be more difficult to see for people with low vision.
  • If you increase the text contrast, then it looks like the input is already filled in, and a user might skip over it accidentally.
  • Some people also use placeholder text to give field hints or formats, but the same issues apply for that as well.

In general, I recommend not using placeholders at all.

Labels must be programmatically defined

Now we've discussed some of the important aspects of visual labelling. But that's not the whole story.

Labels must also be programmatically defined so that assistive technology users also get the full benefits of labels.

What's an accessible name?

The accessible name is the name that assistive technologies communicate to users to identify an element. How does this work (on the web)?

  1. Code
  2. Browser
  3. Operating System (OS) / Accessibility API
  4. Assistive Technology (AT) / User

To start talking about programmatically defined labels, we first have to have an understanding of the accessible name computation.

So what is an accessible name?

All of the explanations I've found tend to be really confusing, so I've tried to strip it down to the main idea.

The accessible name is the name that assistive technologies communicate to users (and vice versa) to identify an element.

By the way, all of these links are available on the slides themselves.

So, how does this work on the web?

  1. First, a developer writes some code.
  2. Then the browser interprets and renders that code.
  3. Then the Operating System, or OS, uses the Accessibility API to send important accessibility information from the browser to the assistive technology, or AT.
  4. Then the AT communicates that information to users.

This can be a complicated process to wrap your head around, but what I want you to take away from it is this: If you, as the developer, don't put accessibility information into the code, it's not going to magically be communicated to the user.

Hopefully this will make more sense in the next few examples.

Labels that aren't programmatically defined

Favorite Animal


<p>Favorite Animal</p>
<input type="text" name="animal">

Here is our first example, where labels aren't programmatically defined.

There's an input with text above it that says "Favorite Animal."

So far, this passes all of our label rules from previous slides. It has a visible, clear, and persistent label. However, in this case, the "label" isn't coded as a <label> element. It's coded as a paragraph element.

There's nothing that programmatically ties this text, "Favorite Animal" with the input field.

Screen reader users who are tabbing through a form will not know what this field is for. They would have to read through the form from top to bottom to find all relevant information.

Similarly, speech-input users might tell their software to click on the "Favorite Animal" input. But speech input software won't recognize this as a named field. So speech-input users will have to find another way to access this field.

Labels that are programmatically defined


<label for="fav-animal">Favorite Animal</label>
<input type="text" name="animal" id="fav-animal">

Now let's look at the same example again, but this time the label, "Favorite Animal," is within a programmatically associated <label>element.

I'm always surprised how many developers I meet, some of them with over 20 years experience, who don't know how to programmatically associate a label. But if that's you, don't worry. I'm going to teach you how now.

So, here in the code we have the label, "Favorite Animal," within a label element. Below is the input.

This alone is not enough to associate the two. To programmatically associate the label to the input, first add a unique id to the input. Then add a for attribute to the label element with a value that matches the value of the id on the input.

In this example, the input has an id value of fav-animal, and the label has a for attribute value of fav-animal.

Now these two elements are programmatically linked, and the label names the field.

Fieldsets in code

<fieldset>
	<legend>Giver</legend>
	<label for="giver-name">Name</label>
	<input type="text" autocomplete="name" id="giver-name">
	<label for="giver-email">Email</label>
	<input type="text" autocomplete="email" id="giver-email">
</fieldset>

ARIA

Accessible Rich Internet Applications

And with that, we've finished our section on HTML labelling and are not moving on to ARIA, or Accessible Rich Internet Applications.

Here I've included an illustration of Merlin the wizard casting a spell over the sword in the stone.

And the reason for that is that people often think of ARIA as magic. Either it's too complex, and they're afraid to touch it, or it's an all-powerful tool that will instantly solve all of your accessibility issues.

Hopefully after this section you'll get a good idea of how to use or not use ARIA.

Before we move on, the first rule of ARIA is that you shouldn't use it at all... unless you have to. Then use it. But if there's an HTML solution, always go to that solution first.

What is ARIA?

WAI-ARIA, the Accessible Rich Internet Applications Suite, defines a way to make Web content and Web applications more accessible to people with disabilities. It especially helps with dynamic content and advanced user interface controls developed with HTML, JavaScript, and related technologies.

Before we talk about how to use ARIA, we should know what it is. According to the WAI-ARIA website (that's W.A.I dash A.R.I.A):

Emphasis mine: WAI-ARIA, the Accessible Rich Internet Application Suite, defines a way to make web content and web applications more accessible to people with disabilities. It especially helps with dynamic content and the advanced user interface controls developed with HTML, JavaScript, and related technologies.

There are two main take aways here. The first is that ARIA is only one way to make things more accessible. There are other solutions as well.

The other is that ARIA is especially useful for dynamic and complex content. That's where it should mostly be used. But there are other occasions, as we'll explore in the next slides, where it's okay to use some ARIA.

Button vs ARIA Button

ARIA Button
            
<button type="button">Button</button>
<div role="button" tabindex="0"
     onclick="doSomething();"
     onkeyup="doSomething();">
  ARIA Button
</div>

Before we dive in, here is my favorite example of why you shouldn't rush to ARIA.

On this page, I have a button element with the text button, and a button made out of ARIA with the text ARIA Button

For the HTML button, the code is a button element type button with the text "Button" inside it. This button is accessible, and there's nothing we need to add to it.

But I see a lot of developers try to make buttons out of divs or links. In this case, there's a div with some ARIA added, in this case, role="button". If it were that simple, that would be great. But ARIA isn't all-powerful. A lot more needs to be added to this div to transform it into a button:

  • It needs to be reachable by keyboard. That means adding tabindex="0".
  • It needs to activate when a user clicks the Spacebar, so there needs to be a key event listener as well as a click event listener.
  • We don't want the Spacebar to scroll the page if it's used to activate the control, so you would need to prevent default
  • You need to change the curser to the expected arrow curser.
  • And if you are using custom focus styles, you'll need to include elements with role="button" or tabindex="0" as well as other interactive elements.

So when you use the proper HTML elements, all the accessibility features are already baked in. If you try to transform one element into another, you need to add a lot more accessibility features to it, and you might accidentally forget some.

Please always use button elements for buttons, and always use the correct HTML element rather than reaching for ARIA.

aria-label

  • One way to provide an accessible name
  • Use where there is no visible text for an element
  • Doesn't work consistently on all elements
  • Will overwrite content: Use caution!
  • Format: aria-label="value"

I think that's enough disclaimers for now, so let's jump into our ARIA labelling solutions.

The first technique we'll look at is using aria-label.

  1. aria-label is one way to provide an accessible name, and that's true of all of our examples today.
  2. Reach for aria-label when there's no visible text on the page that can be used as a label
  3. Some words of caution: Aria labelling doesn't work consistently on all elements, so don't use it on a div or span. Only use it for elements that have an implicit or explicit role. No generic elements.
  4. As we saw in the accessible name computation, adding an aria-label or aria-labelledby will overwrite any existing name. So use aria labelling with extreme caution.
  5. And the format for aria-label is aria dash label equals and the whatever you want the accessible name to be in quotes.

Let's look at an example.

aria-label example

<button type="submit" aria-label="Search">
    ...
</button>

So here is our aria-label example. We have a button with a magnifying glass on it. This button is used with a search form on the Knowbility website.

The magnifying glass is an svg. We're going to use an aria-label to name this button.

So we add aria-label="search" to the button itself. Now this button will be announced by screen readers as a search button.

We don't want to call it magnifying glass because that doesn't make as much sense in context.

aria-labelledby

  • Use to associate visible text with the element
  • Format: aria-labelledby="id1 id2 id3" — space separated id values

Now let's look at aria-labelledby.

The main functional difference between aria-label and aria-labelledby is that you should use aria-labelledby when there is visible text on the page that could name the element.

And the format of aria-labelledby is aria-labelledby (with two L's) equals, and then in quotes, space separated id values. So in this example, I have aria dash labelledby equals "id1 (space) id2 (space) id3"

This should make more sense once we see it in action.

aria-labelledby example


<label for="search" id="search-label">Search</label>
<input type="text" id="search">
<button type="submit" aria-labelledby="search-label">
    ...
</button>

Here we have the same search button, but now there's a form label that says "Search." This label can label both the input and the search button.

To do that, you need to add a unique id to the text element you want to target, in this case the <label>

aria-describedby

  • An accessible description provides additional information, related to an interface element, that complements the accessible name.
  • aria-describedby is one way to provide an accessible description
  • Format: aria-describedby="id1 id2 id3" — space separated id values

Okay, next let's talk about the aria-describedby attribute.

Here's another quote from the W3C. All these links are available in the slides. An aria description provides additional information, related to an interface element, that complements the accessible name.

aria-describedby is one way of adding an accessible description. An accessible description can be provided in addition to an accessible name. It doesn't replace it.

The format for aria-describedby is the same as aria-labelledby. It's aria dash describedby equals and then space separated id values.

aria-describedby example



Must be at least 8 characters 
<label for="pw">Password</label>
<input aria-describedby="pw-reqs" 
       type="text" 
       name="password" 
       id="pw">
<small id="pw-reqs">Must be at least 8 characters</small>

In this example, we have a password field with the label "Password." And then below that are instructions: "Must be at least 8 characters."

A screen reader user tabbing through fields won't encounter these instructions because they're not programmatically associated with the field

Even a screen reader user reading top to bottom would not encounter these instructions until after they fill in the field.

So, we have to programmatically associate them.

To do that, we add a unique id to the element that contains the instructions. In this case, we have a small element. Then you add aria-describedby to the input itself with a value that matches the id value of the instructions.

If this input also had an error message on it, then you could add the id value of the error message in the aria-describedby value separated by a space.

CSS solution: visually-hidden text

  • Use display:none or visibility:hidden to hide content that should be hidden from everyone. Don't rely on opacity, hiding off-screen, or clipped text if the content should be hidden for all users
  • Use a visually-hidden or similar class to hide content visually that adds context for screen reader users

Okay, we're on our last technique. The CSS solution: visually-hidden text

I will say first that if your intention is to hide content from all users, then use display:none or visibility:hidden. This will hide it from everyone, including assistive technology users. If you use opacity, hide things off-screen, or clip text, that content will still be available to assistive technology users and could cause problems.

On the flip side of that, don't use display:none or visibility:hidden to hide things visually but expect screen reader users to access them. They will still be hidden from everyone.

Instead, use a visually-hidden or similar class to hide content visually but add context for screen reader users.

Visually-hidden text example

<button type="submit">
    <span class="visually-hidden">Search</span>
    ...
</button>

And here is our visually-hidden text example.

This is the same search button from earlier. This time instead of using aria-label, we've put some visually-hidden text within the button. We added a span within the button with the class visually-hidden, which will be shown on the next slide. So now the button has the text "Search" within it, but the button still shows only a magnifying glass visually. I prefer this method of adding an accessible name over adding an aria-label, but either way works.

Visually-hidden text code

Code and explanation: The A11y Project.

.visually-hidden {
    clip: rect(0 0 0 0);
    clip-path: inset(50%);
    height: 1px;
    overflow: hidden;
    position: absolute;
    white-space: nowrap;
    width: 1px;
}
